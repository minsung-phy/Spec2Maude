Context and goals
- We worked under the professor-alignment priority:
  - semantic Rule should be emitted as rl/crl
  - evaluation contexts should use heating/cooling rules
  - avoid manual output.maude edits
  - keep CLAUDE.md and STATUS.md aligned
- Main debugging target in this session range was the fib regression/deadlock introduced while pushing toward rl/crl + heating/cooling.

What had already been established before the latest steps
- translator.ml had been moved toward rl/crl generation for Step / Step-pure / Step-read.
- generated label/frame/handler heating/cooling existed in output.maude.
- fib had previously worked in an older mixed setup, but the more isomorphic rl/crl + heating/cooling setup regressed into deadlock.
- a major earlier fix was correcting translator-generated list variable sorts:
  - INSTR-LIST-INSTR and similar variables were changed to WasmTerminals
  - this improved wrapper well-sortedness.

Key findings from this session range
1. The deadlock is not fundamentally caused by exec-loop.
- We removed exec-loop and its manual bridge from wasm-exec.maude.
- We changed fib-body from:
  - CTORBLOCKA2(void-bt, exec-loop(void-bt, fib-loop-body))
  to:
  - CTORBLOCKA2(void-bt, CTORLOOPA2(void-bt, fib-loop-body))
- Result:
  - steps(fib-config(i32v(5))) stopped immediately at the initial block/loop state.
- This showed exec-loop itself was not the core issue.

2. Generated step-read-block/loop do not currently bootstrap the fib harness on their own.
- After removing manual block/loop bridges, these probes were run:
  - CTORBLOCKA2(void-bt, CTORLOOPA2(void-bt, fib-loop-body)) :: Instr
  - CTORLOOPA2(void-bt, fib-loop-body) :: Instr
  - fib-state(i32v(5)) :: State
- Observed results:
  - all reduced to false
- This strongly suggests the generated block/loop rules are not matching the current fib harness term/guard setup directly.

3. Restoring only the manual block/loop bootstrap brings execution back, but deadlock returns as nested labels.
- We restored:
  - step-read-block-manual
  - step-read-loop-manual
- We kept exec-loop removed.
- Result:
  - execution progressed inside the loop again
  - but the system deadlocked in nested restore-label / nested label configurations
  - representative shape:
    restore-label(restore-label(step(< ... >), 0, loop, eps), 0, eps, CTORLOCALGETA1(1))

4. The current strongest diagnosis is:
- manual block/loop bootstrap + generated label heat/cool interact badly
- the outer label heats again around an inner label/restore-label state
- this creates repeated nested label/restore-label states and deadlock

5. A translator-side attempt was made to stop outer label reheating with needs-label-ctxt.
- We modified translator.ml so the auto-generated helper includes:
  - needs-label-ctxt(CTORLABELLBRACERBRACEA3(...)) = true
- Then we rebuilt and regenerated output.maude.
- Result:
  - no improvement in fib
  - steps(fib-config(i32v(5))) still deadlocked
  - modelCheck(fib-config(i32v(5)), <> result-is(5)) still returned counterexample/deadlock
  - modelCheck(fib-config(i32v(5)), [] ~ trap-seen) stayed true
- Additional direct check:
  - needs-label-ctxt(CTORLABELLBRACERBRACEA3(...)) still reduced to false
- So that helper case did not actually affect the runtime behavior as intended.

6. A second translator-side attempt used a dedicated head detector.
- We added:
  - starts-label-ctxt : WasmTerminals -> Bool
- We also changed generated heat-step-ctxt-label so it requires both:
  - needs-label-ctxt(inner) = false
  - starts-label-ctxt(inner) = false
- Then we rebuilt and regenerated output.maude successfully.
- Direct check:
  - starts-label-ctxt(CTORLABELLBRACERBRACEA3(...)) still reduced to false
- Therefore:
  - the issue is not just that needs-label-ctxt lacked a label-wrapper case
  - the current helper-style pattern matching is not recognizing wrapper-headed associative WasmTerminals the way we need

Latest interpretation after those two helper experiments
- The problem is now very likely one of these:
  1. helper-style detection on WasmTerminals is the wrong mechanism for excluding nested label heating
  2. the real fix must be in the generated heat-step-ctxt-label rule shape itself
  3. or the manual block/loop bootstrap must produce a different intermediate label shape so generic label heat/cool does not immediately re-wrap it
