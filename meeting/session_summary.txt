  - disabling all four context rules
    (`ctxt-instrs,ctxt-label,ctxt-handler,ctxt-frame`) still leaves the same overflow
  - conclusion:
    - the overflow is not caused by the baseline direct conditional encoding of the context rules themselves
    - it persists even with all `step-ctxt-*` rules removed

- Baseline stack-overflow root-cause narrowing:
  - `NOP` single-step succeeds in baseline:
    - `step(< fib-state(i32v(1)) | CTORNOPA0 >)` terminates to `eps`
  - singleton `Step-read` examples overflow immediately:
    - `step(< fib-state(i32v(1)) | CTORLOCALGETA1(1) >)`
    - `step(< fib-state(i32v(1)) | CTORGLOBALGETA1(1) >)`
  - helper functions themselves are not the culprit:
    - `$local(fib-state(i32v(1)), 1)` terminates to `i32 0`
    - `$global(fib-state(i32v(1)), 1)` terminates
  - removing `Step/read` and `Step/pure` schema rules did not fix the overflow
  - tightening `step-ctxt-instrs` with a non-empty focus guard (`INSTR =/= eps`) did not fix the overflow
  - a scratch conditional rule with the same `Idx` sort condition and `$local` assignment succeeds immediately
- conclusion:
  - the baseline overflow is not caused by `$local/$global`, `Idx`, or the schema rules alone
  - it is caused by the full direct-conditional `step` rule set itself: Maude's proof search over the overlapping baseline `step` rules explodes even on tiny `Step-read` terms
  - stronger result:
    - the explosion persists even when all baseline `step-ctxt-*` rules are disabled
    - so the direct cause is the full baseline direct-conditional `step` encoding, not heating/cooling replacement via context rules alone

- Focused-step experiment:

Important 2026-04-24 changes:
- Removed the `focused-step` experiment from baseline.
  - No `op focused-step`.
  - No `rl [step-enter]`.
- `output_bs.maude` has no baseline heat/cool context rules.
  - No `heat-step-ctxt-*`.
  - No `cool-step-ctxt-*`.
- `Step/ctxt-instrs` no longer uses synthetic `VALS` / `IS` wrappers.
  - Current shape follows SpecTec more directly:
    `step(< Z | VAL INSTR INSTR1 >) => < ZQ | VAL INSTRQ INSTR1 >`
    with condition:
    `all-vals(VAL) = true /\ step(< Z | INSTR >) => < ZQ | INSTRQ > /\ ...`
- Scalar type guards are now preserved for variables that appear in the generated rule text, including RHS and premise-result variables.
  - Example: `Step/ctxt-frame` now generates `SQ : Store` and `FQQ : Frame`.
  - Example: `Step/ctxt-instrs` now generates `ZQ : State`.
- Sequence `hasType(list(instr))` guards are still intentionally omitted in baseline.
  - Reason: they are expensive over associative sequence terms and are not currently required for the scalar-guard correctness issue.
  - `all-vals(VAL) = true` remains because it represents the SpecTec `val*` side condition.

Verification:
- `dune build ./main_bs.exe` succeeds.
- `dune exec ./main_bs.exe -- wasm-3.0/* > output_bs.maude` succeeds.
- `maude -no-banner output_bs.maude` loads.
- Loading still emits existing warnings/advisories for some generated rules; this session did not attempt to eliminate every warning.

Baseline stack-overflow result:
- The direct conditional baseline still stack-overflows for fib model checking.
- Prior narrowing results:
  - `NOP` singleton step succeeds.
  - `LOCAL.GET` and `GLOBAL.GET` singleton `Step-read` examples stack-overflow under the full baseline rule set.
  - `$local(fib-state(i32v(1)), 1)` and `$global(...)` reduce normally by themselves.
  - A scratch rule with similar `Idx` and `$local` conditions succeeds.
  - Disabling `step-ctxt-instrs`, `step-ctxt-label`, or all four `step-ctxt-*` context rules does not remove the overflow.
  - The removed `focused-step` indirection also did not remove the overflow.
- Current conclusion:
  - The direct cause is not `$local`, `$global`, `Idx`, or the context rules alone.
  - The full direct-conditional executable `step` relation causes Maude's conditional rewrite proof search to explode over overlapping rules, even for tiny concrete `Step-read` terms.
