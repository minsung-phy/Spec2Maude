    **WASM 연구 얘기**
    
    **1. 문제의식 및 배경**
    
    - WASM(WebAssembly)의 Semantics는 이미 정의되어 있음.
    - 그러나:
        - 문서로 기술된 Semantics는 **버전/문서마다 표현이 달라짐**
        - 정의는 존재하지만 **정확성/일관성에 대한 의문이 지속적으로 제기됨**
    - 여러 연구 그룹에서
        - **프로그램 Semantics를 언어 시스템 코드처럼 형식적으로 기록**하려는 시도가 있어 왔음
    - **K Framework**는 이를 위한 대표적인 프레임워크 중 하나
        - 도메인 특화 언어(DSL) 기반
        - 커뮤니티와 긴밀히 협업
        - WASM Semantics도 연구 대상 (KAIST 쪽 연구 포함)
    
    ---
    
    **2. Spectec과 Rewriting Semantics**
    
    - **Spectec**
        - WASM 공식 Semantics 명세 언어
        - 내부적으로 **rewriting 기반** Semantics
    - Spectec과 Maude/K의 관계
        - Semantics 개념 자체는 **약 80% 유사**
        - 하지만:
            - 그대로 100% 옮길 수는 없음
            - **사용 방식과 추상화 수준에서 약 20% 차이**
    - 핵심 아이디어
        - **Spectec Semantics를 Maude/K로 변환**
        - 기존 Spectec 정의를 기반으로 의미론을 재구성
    
    ---
    
    **3. 기술적 난점**
    
    - Spectec ↔ Maude/K **양쪽을 모두 이해해야 함**
        - 이 과정에서 시행착오가 있었음
    - 현재 접근
        - IL → Parse Tree 분석
        - 이를 기반으로 **Maude 정의 자동 생성**
    - 현황
        - 변환 자체는 **약 50% 진행**
        - 하지만 **실행이 안정적으로 동작하지 않음**
        - 의미론 정의 ≠ 실행 가능 의미론이라는 문제가 존재
        - K의 Heating & Cooling rules를 사용하면 잘될듯
    
    ---
    
    **4. 모델 체킹(Model Checking) 관점**
    
    - 일반적으로 모델 체킹은
        - **분산 시스템**, 통신 기반 시스템에서 많이 사용됨
    - 웹 애플리케이션 역시:
        - 컴포넌트 간 **상호작용(communication)**이 핵심
    - 목표
        - **애플리케이션 전체를 대상으로 모델 체킹**
    - 문제
        - **상태 폭발(State Explosion)** 발생
    - 가능성 있는 해결 방향
        - 기존과 다른 **deduction 기반 알고리즘**
        - 더 효과적인 상태 축소/추론 기법
    
    ---
    
    **5. 연구 포인트 및 기회**
    
    - 현재 상태:
        - **이 문제를 본격적으로 해결한 연구는 거의 없음**
    - 연구 기회:
        - 새로운 모델 체킹 기법 + WASM/K Semantics 결합
        - **PLDI에 논문 내자**
    - 진척이 크지 않아 공백 상태
    
    ---
    
    **6. 향후 방향**
    
    - 단순 변환이 아닌:
        - **실행 가능한 시맨틱스 확보**
    - 적절한 규모의 애플리케이션을 대상으로:
        - 효과적인 분석 테크닉 개발
    - **사례 연구(case study)** 중심으로 검증
    - K Framework 기반 의미론을 활용한:
        - 실질적인 모델 체킹 가능성 탐구

    **의문점**
    
    1. **EL을 변환하는게 아니라 IL을 변환하는거라면 spectec/specification/wasm-3.0/**.spectec을 변환하는게 아니라 spectec/src/../*.ml을 변환하는거겠네 ?**
        
        **그러면 ml 코드를 자동으로 변환하면 될텐데 AST를 보고 변환한다는게 무슨 소리인지 이해가 잘 안가**
        
        - ㄴㄴ → *.ml은 spectec을 받아서 ast(= IL)로 만드는 인터프리터임
        - 그러면 나는 ast(= IL)을 Maude로 바꾸는 변환기(ml 코드)를 만들어야함
        
    2. **어떻게 자동으로 변환을 해야하는지 아이디어가 아예 생각이 안나 ..**
        - 구현 흐름 (Implementation Flow)
            1. **기존 상태 (SpecTec):**
                - `main.ml` 등의 진입점이 `.spectec` 파일을 읽습니다.
                - `parser.ml` 등이 이를 파싱하여 `AST` 데이터 구조(메모리 상의 객체)를 만듭니다.
                - 기존 로직은 이 AST를 가지고 인터프리터 실행을 하거나 타입 체크를 합니다.
            2. **내가 해야 할 작업 (Action Item):**
                - SpecTec 소스 코드(`src/`) 내에 새로운 파일(예: `maude_backend.ml`)을 만듭니다.
                    - 이 파일 안에서 **"AST를 입력받아 Maude 문법의 문자열(String)을 뱉어내는 함수”**를 OCaml로 짭니다.
                - `main.ml`을 수정하여, 프로그램 실행 시 이 함수가 호출되도록 연결합니다.
        - 예시로 이해하기 (Example)
            - **입력 (`test.spectec`) :** `num : 42`
            - **Step 1: 기존 `*.ml`이 만드는 AST (Data):** (OCaml의 데이터 타입으로 메모리에 존재)
                
                `Const (Int 42)`
                
            - **Step 2: 귀하가 작성해야 할 OCaml 코드 (Logic):**
                
                `(* 귀하가 작성할 maude_backend.ml *)
                let rec to_maude (node : ast) =
                  match node with
                  | Const (Int n) -> "op " ^ (string_of_int n) ^ " : -> Nat ."
                  | ...`
                
            - **Step 3: 결과물 (`test.maude`):**
                
                `op 42 : -> Nat .`

                ```jsx
                syntax binop_(numtype)
                syntax binop_(Inn) =
                  | ADD | SUB | MUL | DIV sx hint(show DIV#_#%) | REM sx hint(show REM#_#%)
                  | AND | OR | XOR | SHL | SHR sx hint(show SHR#_#%) | ROTL | ROTR
                syntax binop_(Fnn) =
                  | ADD | SUB | MUL | DIV | MIN hint(macro "FMIN") | MAX hint(macro "FMAX") | COPYSIGN
                ```
                
            - 1.3-syntax.instructions.maude :
                
                ```jsx
                sort WasmTerminal .
                sort WasmType .
                
                op binop_ : WasmTerminal -> WasmType .
                op sx : -> WasmType .
                
                op ADD : -> WasmTerminal .
                op SUB : -> WasmTerminal .
                
                op typecheck _ _ : WasmTerminal WasmType -> BOOL .
                var k : WasmTerminal .
                eq typecheck(ADD, binop_(k)) = true
                	if typecheck(k, Inn)
                ```
                
                - 타입 하나하나 정의할 수 없으니까 WasmType, WasmTerminal 선언해서 typecheck를 하는게 tip임

    ### **현재 아키텍처에 대한 이론적 피드백 (Coherence & Confluence)**
    
    - **교수님 평가 :** 룰의 중복을 줄이고 명세서의 순수성을 살리려는 엔지니어링적 접근(투-티어)은 좋았음
    - **문제점 지적 :** Maude에서 상태 전이를 담당하는 래퍼(Wrapper)에 방정식(`eq`, `ceq`)과 룰(`crl`, `rl`)을 섞어 쓰는 것은 수리논리학적으로 매우 위험함
    - **이유 :** 방정식(`eq`)은 '언제든 계산해도 결과가 같음'을 전제함(Confluence & Termination). 만약 상태가 변하는 환경에서 룰과 방정식이 꼬이면, Maude 엔진이 가능한 실행 경로를 놓쳐버리는 **코히어런스(Coherence) 붕괴 문제**가 발생함.
    - **결론 :** 룰의 왼쪽(LHS)에는 리듀스(Reduce)되는 함수가 오면 안 되며, 오직 데이터를 담는 **컨스트럭터(Constructor)**만 와야 수학적으로 안전함.
    
    ### 핵심 이슈 : SpecTec과 Maude의 '의미론(Semantics) 불일치'
    
    - **SpecTec의 실행 방식 :** Evaluation Context 기반. 명령어는 반드시 스택의 '맨 위(Top-most)' 또는 정해진 문맥에서만 실행되어야 함.
    - **Maude의 실행 방식 :** Arbitrary Position Rewriting. 엔진이 텀(Term)의 중간이든 밑바닥이든 패턴이 매칭되면 임의의 위치에서 룰을 실행해 버림.
    - **문제점 :** 만약 컨텍스트(`STATE`, `VALS`)를 제거하고 알맹이 룰만 남겨두면, Maude 엔진이 스택 중간에 있는 명령어를 먼저 계산해버려 Wasm 시맨틱이 완전히 깨지게 됨.
    
    ### 해결 방안: K-Framework 방법론 도입 (`[frozen]` & Heating/Cooling)
    
    독자적인 래퍼 방식 대신, 정형 검증 학계의 스탠다드한 방법론(K-Semantics)을 Maude로 구현하여 적용할 것.
    
    1. **`[frozen]` 속성 적용:** 컨텍스트(스택, 상태 등)를 묶는 오퍼레이터에 `[frozen]` 속성을 부여.
        - 엔진이 스택 중간을 자기 맘대로 파고들어 계산하지 못하도록 '얼려두는' 역할.
    2. **Heating / Cooling 룰 도입:**
        - **Heating:** 얼어있는 전체 컨텍스트에서 '현재 실행할 수 있는 명령어(Top-most)'만 찾아내어 밖(실행 가능한 위치)으로 뽑아냄.
        - **Execution:** 뽑아낸 명령어를 순수 연산 룰(`rl`)로 계산함.
        - **Cooling:** 계산된 결과값을 다시 원래 얼어있던 컨텍스트의 빈자리(Hole)에 돌려놓음.

    ## 1. K-Framework 적용의 개념적 오류
    
    - **정적 구조와 동적 태스크의 혼동:** K-Framework의 아이템(`~>`)은 컴퓨테이션 태스크(동적인 계산 흐름)를 의미함. Wasm의 스택(Stack)과 같은 정적인 데이터 구조를 `~>`로 나열하는 것은 이론적으로 잘못된 접근임("타이어를 몸에 두르고 다니는 격").
    - **`frozen` 속성 사용의 모순:** K-Framework 이론 자체에는 `frozen`이라는 개념이 존재하지 않음. 따라서 K-Framework를 사용한다고 하면서 Maude의 `frozen` 속성으로 리라이팅을 통제하려는 것은 앞뒤가 맞지 않는 방식임.
    - **올바른 K-Framework(Decomposition)의 활용:** 커다란 문제를 디컴포즈(Decomposition)하여 실행 가능한 단위로 쪼개고(Heating), 서브 태스크가 완료되면 다시 원래 컨텍스트로 복구(Cooling)하는 개념을 명확히 수식적으로 정의해야 함.
    
    ## 2. 시맨틱의 차이: SpecTec (1-Step) vs Maude (0 or More Steps)
    
    - **근본적인 실행 단위의 차이:** SpecTec의 룰은 정확히 원 스텝(One-step)의 상태 전이를 의미함. 반면, Maude의 리라이팅 로직은 임의의 서브 텀 위치에서 0번 이상의 스텝(Zero or more steps)으로 동시다발적으로 발생할 수 있음.
        
        <aside>
        💡
        
        ### "1 Step" vs "0 or More Steps"의 의미
        
        이 개념은 프로그램이 "어떻게 실행되는가(시맨틱)"를 묘사하는 방식의 차이야.
        
        - **SpecTec (Structural Operational Semantics, 1 Step):**
            - SpecTec의 규칙은 **딱 한 번의 계산(One-step transition)**만 정의해.
            - 예를 들어, `(CONST 1) (CONST 2) ADD => (CONST 3)` 이라는 규칙이 있다면, 이 규칙을 한 번 적용하면 덧셈 딱 한 번만 하고 멈춰야 해. "다음 명령어가 뭔지"는 이 규칙의 관심사가 아니야. 오직 "현재 상태에서 다음 상태로 가는 딱 한 걸음"만 서술해.
        - **Maude Rewriting Logic (0 or More Steps):**
            - Maude의 리라이팅(`=>`)은 **"조건만 맞으면 엔진이 알아서, 될 때까지, 어디서든 계속 바꾼다"**는 뜻이야.
            - 만약 Wasm 스택에 `(CONST 1) (CONST 2) ADD (CONST 4) (CONST 5) ADD`가 있다고 치자.
            - 우리가 원하는 순서는 왼쪽 `ADD`를 먼저 해서 `(CONST 3)`을 만들고, 그 다음에 오른쪽 `ADD`를 하는 거야. (1스텝씩 차례대로).
            - 하지만 Maude 엔진에 덧셈 규칙(`ADD => 결과`)을 그냥 던져주면, 엔진은 "어? 왼쪽 `ADD`도 바꿀 수 있고, 동시에 오른쪽 `ADD`도 바꿀 수 있네? 둘 다 한꺼번에 바꿔버려야지!" 하고 지맘대로 순서를 무시하고 동시다발적으로 계산을 진행해 버려. 심지어 룰 적용이 끝날 때까지 (더 이상 바꿀 게 없을 때까지) 멈추지 않아.
        
        **결론:** SpecTec은 "이 톱니바퀴는 한 칸만 돌아간다"라고 명세했는데, Maude에 그대로 번역해 넣으면 "이 톱니바퀴는 돌릴 수 있을 때까지 미친 듯이 다 돌아간다"로 의미가 왜곡되는 거야. 이 차이 때문에 "잘못된 실행 흐름"이 발생한다고 교수님이 지적하신 거야.
        
        </aside>
        
    - **`step` 오퍼레이션 도입의 필요성:** 단순히 SpecTec 룰을 Maude 리라이팅 룰로 1:1 변환하면 이 시맨틱 차이로 인해 잘못된 실행 흐름이 발생함. 이를 통제하기 위해 명시적으로 `step`이라는 오퍼레이션을 정의하여 한 스텝만 실행되도록 강제하는 방식을 고려해야 함.
        
        <aside>
        💡
        
        **`step` 오퍼레이션이 어떻게 이 문제를 해결하는가?**
        Maude가 제멋대로 룰을 연속해서 적용하는 걸 막으려면, 엔진에게 **"지금은 딱 한 번만 룰을 적용해!"**라고 목줄을 채워야 해. 그 목줄이 바로 `step` 오퍼레이션이야.
        • **해결 원리:**
            1. 상태 $S$ 전체를 `step(S)`라는 특별한 껍데기(오퍼레이션)로 감싸.
            2. 리라이팅 룰을 정의할 때, 조건으로 "반드시 겉에 `step(...)`이 씌워져 있을 때만 실행해라"라고 제약을 걸어.
            3. 룰이 한 번 적용되어서 상태가 $S'$로 바뀌면, **그 결과물에는 `step` 껍데기를 벗겨서 출력**하도록 규칙을 만들어. (예: `step(S) => S'`)
        • **효과:**
            ◦ 처음에 `step(S)`로 시작하면 룰이 딱 한 번 적용돼서 `S'`가 돼.
            ◦ `S'`에는 더 이상 `step` 껍데기가 없지? 그러니까 Maude 엔진이 아무리 더 바꾸고 싶어도 "어? `step` 껍데기가 없네? 더 이상 룰 적용 못 하겠다." 하고 얌전히 멈추게 돼.
            ◦ 이 방식으로 Maude의 "0 or More Steps" 폭주를 완벽하게 "1 Step"으로 통제할 수 있어. `frozen` 같은 꼼수 없이 수학적으로 아주 깔끔하게 말이야.
        
        </aside>
        
    - **참고 자료:** 이전에 수행했던 Structural Operational Semantics (Lec 5) 과제에서 스몰 스텝(Small-step)을 정의할 때 `step` 연산자를 어떻게 활용했는지 복기할 것.
    
    ## 3. 조건부 룰(`crl`)의 문제점과 무조건부(`rl`) 변환의 당위성
    
    - **모델 체킹(Model Checking)의 분석 한계:** LTL 모델 체킹은 상태 간의 전이(Transition) 결과만 추적함. 전이가 일어날 때 조건절(`if ...`) 내부에서 수행되는 세부적인 디덕션(Deduction) 과정은 모델 체커가 분석하지 못하고 무시함.
    - **치명적인 성능(Performance) 저하:** 조건부 룰(`crl`)은 매칭 시 조건이 참인지 판단하기 위해 수많은 내부 트랜지션(Exhaustive check)을 발생시켜 퍼포먼스를 기하급수적으로 떨어뜨림(예: 혁순 선배의 KCC 구현체에서 1분 이상 소요되는 현상).
    - **해결 방향:** 특정 컨디셔널 룰을 어떻게 언컨디셔널 룰(Unconditional rule, `rl`) 형태로 트랜스포메이션할 것인지 방법론을 고안해야 함.

    2. crl을 쓰지말라고 한 부분은 Evaluation Context 처리 부분임 (label, frame, instrs, handler)
        
        → heating/cooling rule로 처리
        
        ```jsx
        rule Step/ctxt-label:
            z; (LABEL_ n `{instr_0*} instr*)  ~>  z'; (LABEL_ n `{instr_0*} instr'*)
            -- Step: z; instr* ~> z'; instr'*
        ```
        
        ```jsx
        S--- 1. 복구를 위한 보조 operator 선언 (f의 역할)
        op restore-label : Config Nat InstSeq InstSeq -> Config .
        
        --- 2. Heating : Label을 만나면 step을 껍데기 안쪽(INSTRS_INNER)으로 밀어넣음
        rl [heat-label] :
        	step( (S ; F) ; (LABEL_ n {INSTRS_0} INSTRS_INNER) INSTRS )
        	=>
        	restore-label( step( (S ; F) ; INSTRS_INNER ) , n, INSTRS_0, INSTRS ) .
        
        --- 3. Atomic Step (실제 실행): 안쪽 명령어가 1스템 실행되며 step 껍데기를 소모함
        --- 예: rl [step-pure-binop] : step(...) => (S' ; F') ; INSTES_INNER' .
        
        --- 4. Cooling : 안쪽 명령어가 1스템 실행되어 순수 Config로 돌아오면 다시 LABEL을 씌움
        rl [cool-label] :
        	restore-label( (S' : F') ; INSTRS_INNER', n, INSTRS_0, INSTRS )
        	=>
        	(S' ; F') : (LABEL_ n {INSTRS_0} INSTRS_INNER') INSTRS .
        ```
