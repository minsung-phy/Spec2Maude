# Footer `= valid` Leftovers Audit

Updated: 2026-05-20

## Scope

This audit classifies and closes the footer `eq` / `ceq ... = valid` statements
that remained after strict validation lowering reached:

```text
281 strict source-rule targets
-> 281 primary Maude rl/crl rules
```

Strict C1 policy does not allow duplicate source-rule footer equations. If a source relation rule already has a primary generated `rl` / `crl`, a footer equation for the same source fact is non-isomorphic duplicate scaffolding.

## Generator Location

The footer is generated by the `footer` string in `translator_bs.ml`.

The sequence-shaped `Val-ok` list-lifting block used to live in that footer near
the frame/prelude helper block. It has now been removed from strict C1.

Current source-generated singleton relation declarations/rules remain untouched:

- `op Val-ok : WasmTerminal WasmTerminal WasmTerminal -> Judgement [ctor]`
- primary `crl [val-ok-*]` rules generated from `4.1-execution.values.spectec`

## Rationale

`Expand`, `Num-ok`, and singleton `Val-ok` are real SpecTec source relation rules. They are already emitted by generic RelD lowering as primary Maude `crl` rules, so the footer equations were duplicate source-rule scaffolding.

The sequence-shaped `Val-ok` equations were different: SpecTec has singleton
`Val_ok`, while some execution premises use pointwise iteration over `val*` /
`valtype*`. The footer equations provided an executable list-lifting shortcut
for that harness/prelude use. They are not strict C1-final.

A focused ablation removed only this footer overload/equation pair from the
generator and regenerated `output_bs.maude`. Accepted execution smokes still
passed:

- `$expanddt(value('TYPE, fib-funcinst))`;
- label/br + suffix search;
- br_if + suffix search;
- nop + suffix search;
- `steps(fib-config(i32v(5)))`.

The value-validation probes now classify the removed block precisely:

- `Val-ok(fib-store, CTORCONSTA2(CTORI32A0, 5), CTORI32A0)` still rewrites to
  `valid` through the source-generated singleton rule.
- `Val-ok(fib-store, eps, eps)` remains stuck.
- `Val-ok(fib-store, CTORCONSTA2(CTORI32A0, 5) CTORCONSTA2(CTORI32A0, 0),
  CTORI32A0 CTORI32A0)` remains stuck.

This confirms the removed equations were only executable list-validation
scaffolding, not required by the accepted Fibonacci execution smokes.
