## 1. 프로젝트의 전반적인 목표

### 1.1 연구 목표 (큰 그림)

- WebAssembly 명세(SpecTec)를 Maude로 자동 변환하고, 이를 이용해 실행 의미론 검증과 모델체킹을 수행한다.
- 단순 실행기 제작이 아니라, SpecTec 명세를 재활용 가능한 형태로 분석/검증하는 연구 파이프라인을 만든다.
- 최종적으로는 사례(예: fib 및 추가 벤치마크)에서 속성 검증을 통해 의미론적 문제를 찾을 수 있어야 한다.

### 1.2 엔지니어링 목표 (구현 관점)

- 입력: SpecTec EL을 정제한 IL AST
- 출력: Maude 모듈(자동 생성 규칙 + 실행/검증 가능 형태)
- 핵심: 하드코딩된 개별 규칙 나열이 아니라, 패턴 기반의 일반 변환 로직 확보

### 1.3 일반화 목표 (확장 관점)

- 타깃은 "Wasm 하나만 겨우 동작"이 아니라 "SpecTec 계열 언어 전반으로 확장 가능한 변환기"다.
- 즉, Wasm 전용 임시 우회보다 일반적인 Evaluation Context 변환 전략이 우선이다.

## 2. 교수님 요구사항 핵심 요약

## 2.1 의미론 보존이 최우선

- SpecTec 규칙을 Maude로 옮길 때 "모양이 비슷한 것"이 아니라 "실행 의미가 같은 것"을 보장해야 한다.
- SpecTec은 Top-most/Context 기반 1-step 의미인데, Maude는 임의 서브텀 위치에서 0-or-more rewrite가 가능하므로 그대로 옮기면 의미가 깨질 수 있다.

## 2.2 Eq/Ceq와 Rl/Crl 혼용 리스크 관리

- 상태 전이 영역에서 eq/ceq와 rl/crl을 무분별하게 섞는 방식은 coherence/confluence 리스크가 크다.
- 룰 LHS에는 reducible function symbol이 아니라 constructor 중심 패턴을 유지해야 안전하다.

## 2.3 Context 규칙은 Heating/Cooling으로 처리

- label/frame/instrs/handler 같은 Evaluation Context는 별도 heating/cooling 규칙으로 분해/복원해야 한다.
- 컨텍스트를 단순 생략하거나 "중간 스택에서도 매칭"되게 두면 잘못된 실행이 발생한다.

## 2.4 Step 단위 강제

- SpecTec의 1-step 의미를 보존하기 위해 step 연산자 기반의 통제가 필요하다.
- "룰 하나 적용 = 정확히 한 스텝"이라는 운영 모델이 유지되어야 한다.

## 2.5 Conditional 룰 남용 금지

- 컨디셔널 내부 계산은 모델체킹 관점에서 추적성이 낮고, 성능 저하를 크게 유발한다.
- 따라서 가능한 영역은 unconditional화(또는 조건 최소화)하고, 정말 필요한 조건만 남겨야 한다.
- 단, RHS 함수 실패 가능성이 있는 경우에는 안전 가드가 있는 crl 형태가 필요할 수 있다.

## 2.6 K-Framework "직접 적용"과 "아이디어 활용" 구분

- 이 프로젝트는 K 정의를 새로 짜는 과제가 아니다.
- K의 핵심 아이디어(Decomposition, Heating/Cooling)는 활용하되, SpecTec -> Maude 변환 목표를 벗어나지 않아야 한다.

## 3. 이 프로젝트의 비목표 (하지 말아야 할 방향)

- K configuration 자체를 새로 정의하는 별도 K 프로젝트로 확장
- Wasm 특정 예제만 통과시키는 과도한 하드코딩
- 의미 보존 검토 없이 단순 텍스트 치환식 변환
- 모델체킹에 불리한 거대 conditional chaining 의존

---

## 4. 현재 기준 구현 원칙

### 원칙 1. "정확성 > 편의성"

- 당장 짧게 돌아가는 우회보다 의미 보존이 우선이다.

### 원칙 2. "일반 규칙 > 예외 하드코딩"

- 특정 명령어 이름 하드코딩보다 IL 구조 기반 패턴 변환을 우선한다.

### 원칙 3. "Context 분해 명시화"

- context 관련 규칙은 heating/cooling으로 분리해 추적 가능하게 유지한다.

### 원칙 4. "검증 가능성 내장"

- 변환 결과는 rewrite와 modelcheck에서 즉시 검증 가능한 형태여야 한다.

### 원칙 5. "문서와 코드 동기화"

- 문서의 변환 규칙 설명과 translator 코드 경로가 항상 1:1로 대응되어야 한다.

---

## 5. 최종 산출물 기준

최소한 아래 3축이 동시에 만족되어야 한다.

- 변환기 품질: IL AST -> Maude 자동 변환이 일반 패턴에서 안정적으로 동작
- 의미론 품질: SpecTec 의미를 깨지 않는 step/context 처리
- 검증 품질: rewrite/modelCheck 결과가 재현 가능하고 문서화됨

이 기준이 충족되어야 "SpecTec 명세의 자동 변환 + 검증 가능한 Maude 실행"이라는 프로젝트 목표를 달성한 것으로 본다.
