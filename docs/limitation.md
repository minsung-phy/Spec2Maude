# C1 Strict Limitations

Updated: 2026-05-20

This is the single current summary for the C1 strict baseline. The detailed
files under `docs/c1-*` are evidence/archive material. For day-to-day work,
read this file first, then `STATUS.md` for commands and handoff notes.

## Current C1 Structural State

The generated strict baseline is `output_bs.maude`, produced by
`translator_bs.ml`.

Confirmed source coverage:

- syntax declarations: 249 / 249;
- def declarations/equations: 1272 / 1272;
- relation declarations: 82 / 82;
- rule declarations: 499 / 499;
- missing source constructs: 0.

Strict validation lowering is structurally complete:

- 281 / 281 strict validation source-rule targets are primary Maude
  `rl` / `crl`;
- no source validation target remains as `eq` / `ceq ... = valid`;
- no `eq` / `ceq ... = valid` rows remain in `output_bs.maude`;
- no `iter-empty` / `opt-empty` derived validation rules remain.

Under the current project definition of C1 isomorphism, representation names may
differ from SpecTec source names when the source structure and intent are
preserved. The remaining strict non-isomorphic execution debt is the 20
label-related `step-from-step-pure-*` shortcuts described below.

## What Is No Longer A Limitation

These were previously suspicious or non-C1-final, but are now resolved or
accepted:

- `_shape-x_` was a dead extra header declaration. It was removed from
  `translator_bs.ml` and regenerated `output_bs.maude`. The source shape syntax
  `syntax shape = lanetype X dim` is represented by generated `CTORXA2`, with
  source-derived `$dim`, `$lanetype`, and `$unpackshape` equations.
- The footer `$expanddt` shortcut was removed. Only the source-generated
  `$expanddt` definition remains.
- The source-rule footer duplicates for `Expand`, `Num-ok`, and singleton
  `Val-ok` were removed.
- The sequence-shaped footer `Val-ok` list-lift was removed. Empty/multi-value
  `Val-ok` probes are now documented as strict executability limitations rather
  than hidden by footer equations.
- `is-val` / `all-vals` were removed. Source `val*` premises now use the
  generated category-sequence predicate `$is-spectec-val-seq`, derived from the
  SpecTec `val` category.
- `$mk-frame` was removed. Manual tests should use the source-derived frame
  constructor `CTORFRAMEA2(locals, moduleinst)`.
- `$local`, `$with-local`, and `$with-locals` are source-derived and should not
  be treated as helper debt.
- `$subst-typeuse`, `$subst-valtype`, and `$subst-subtype` sequence lifts are
  accepted C1 representation substrate for SpecTec star-map notation such as
  `f(x)*`. They should eventually be generated by a generic star-map lowering
  path instead of handwritten footer strings, but they are not currently treated
  as C1 isomorphism violations.
- `StepConf`, `StepPureConf`, `StepReadConf`, and `StepsConf` are accepted
  Maude relation-compilation substrate for this project, not current
  non-isomorphic debt.
- `$rec-typevars`, `$def-typeuses`, `$idx-typeuses`, `$cfg-state`,
  `$cfg-instrs`, `needs-label-ctxt`, `is-trap`, stale `VALOK-*` variables, and
  the disabled `ExecConf restore-*` branch were removed.

## 1. Label-Related `step-from-step-pure-*` Debt

Remaining non-C1-final scaffold:

- 20 label-related `step-from-step-pure-*` rules remain in `output_bs.maude`;
- non-label `step-from-step-pure-*` shortcuts are absent.

These rules are derived Step_pure-to-Step shortcuts. They are not direct
SpecTec source rules and should not be considered C1-final.

Why they remain:

- the strict `Step/ctxt-instrs` rule is structurally close to the source rule;
- direct `step-pure(label(... br 0)) => eps` succeeds;
- but without the label shortcuts, `step((Z ; label(... br 0)))` does not
  operationally arise through the generic `step-pure` bridge;
- therefore the suffix case `step((Z ; label(... br 0) local.get 1))` also
  fails, because `Step/ctxt-instrs` needs that inner Step premise.

Attempts already ruled out:

- condition reordering caused runaway recursion / stack overflow;
- broadening the bridge result to `StepPureConf` admitted zero-step unreduced
  `step-pure(...)` terms and was not C1-faithful;
- Maude 3.5.1 does not accept a non-reflexive `=>!` condition form for this
  use.

Professor-facing question:

> Should C1 keep these 20 label-related executable shortcuts as documented
> debt, or should they move to C2 while C1 remains structurally faithful but
> less executable?

## 2. Empty `*` Premise Limitation

Examples:

- `Resulttype-ok(C, eps)`;
- `Resulttype-sub(C, eps, eps)`;
- `Instrtype-ok(C, arrow(eps, eps, eps))`;
- `Instrtype-sub(C, arrow(eps, eps, eps), arrow(eps, eps, eps))`.

Strict C1 preserves the source iterated premise structurally. Without derived
`iter-empty` rules, plain Maude rewriting does not discharge some empty-list
cases. Reintroducing `iter-empty` / `opt-empty` rules would improve
executability, but those rules are derived executable scaffolding and are not
allowed in strict C1.

## 3. `Instrs-ok/seq` Witness Synthesis Limitation

Example:

- `Instrs-ok(C, CTORNOPA0, arrow(eps, eps, eps))`.

The source rule has an intermediate witness variable, such as `TS2`, that is
produced by one premise and consumed by another:

```text
Instr-ok(C, instr1, arrow(TS1, XS1, TS2)) => valid
Instrs-ok(C', instrs2, arrow(TS2, XS2, TS3)) => valid
```

The current `Judgement => valid` encoding checks relation facts but does not
synthesize such intermediate witnesses. This is a mode / witness-search
limitation, not a missing primary rule.

A related strict-execution probe also showed that sequence indexing such as:

```text
index(value('LOCALS, C), eps)
```

does not reduce in the current generic prelude. A list-lifted sequence-index
equation could make this more executable, but that is solver/prelude design
work, not a missing SpecTec source rule.

## 4. Sequence `Val-ok` Executability Limitation

Examples:

- `Val-ok(fib-store, eps, eps)`;
- `Val-ok(fib-store, CONST i32 5 CONST i32 0, i32 i32)`.

Strict C1 keeps only the source-generated singleton `Val-ok` rules, such as
`Val-ok/num` and `Val-ok/vec`. The previous footer equations that lifted
`Val-ok` pointwise over `val*` / `valtype*` were executable scaffolding and
have been removed from the strict core.

The singleton probe:

```text
Val-ok(fib-store, CTORCONSTA2(CTORI32A0, 5), CTORI32A0)
```

still rewrites to `valid`. Empty and multi-value sequence probes remain stuck
unless a later C2-style execution layer or principled mode-aware validation
solver provides list lifting.

## 5. Concrete Store / Harness Lookup Limitation

Example:

- `Externaddr-ok/func` with `fib-store`.

The generated source rule needs:

```text
index(value('FUNCS, s), a)
```

to expose a function instance and project `TYPE`. In the current Fibonacci
harness, the lookup itself succeeds:

```text
index(value('FUNCS, fib-store), 0) = fib-funcinst
```

but `fib-funcinst` is an opaque harness constant with field-projection
equations. The generated structural predicate `$is-spectec-funcinst` recognizes
the source record shape for function instances and does not reduce on that
opaque constant. This is a concrete harness representation limitation, not a
missing source-rule lowering.

## 6. Footer / Prelude / Genericity Debt

The strict core is cleaner now, but the header/footer/prelude still need a
later genericity pass. Current debt is about organization and generalization,
not missing Wasm source coverage.

Known items:

- carrier names such as `WasmTerminal`, `WasmTerminals`, `WasmType`, and
  `WasmTypes` are accepted for the current Wasm C1 but should become generic or
  parameterized before non-Wasm specs such as P4;
- fixed Wasm index sorts and atoms in the header should eventually be generated
  from source declarations rather than maintained as fixed strings;
- `$subst-*` star-map sequence lifting should be generated generically, even
  though it is accepted as C1 representation substrate;
- `CTORFRAMEA2` is source-derived for Wasm frame syntax, but the typed
  record-constructor pattern should eventually be generated generically from
  source record syntax.

## 7. Source-Module / Init-Config Limitation

The professor-facing story cannot remain only:

```text
manual fib-store / fib-funcinst / fib-moduleinst
  -> hand-assembled runtime config
  -> steps(...)
```

The desired later story is:

```text
wasm / wat code
  -> Wasm module Maude term
  -> validation-preserving initial config generation
  -> generated C1 semantics
  -> steps / search / verification
```

This is intentionally not solved in the current C1 cleanup pass.

## 8. Model Checking Note

C1 strict is a structural / isomorphic baseline. It is not intended to be an
analysis-optimized semantics.

Model checking, mode-aware validation solving, witness synthesis, and
execution-oriented control infrastructure should be treated as C2-or-later work
unless explicitly accepted into C1 after review.
